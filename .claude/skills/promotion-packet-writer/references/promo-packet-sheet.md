# Promo Packet Sheet

Detail for `promotion-packet-writer`. Read on demand.

## Rubric-dimension map (typical)

| Dimension | The case must show |
|---|---|
| Scope | Breadth/complexity of what was owned (team → org → company) |
| Technical | Depth, judgment, hard problems solved |
| Leadership / influence | Multiplying others, direction-setting, mentorship outcomes |
| Impact | Measurable business/user/eng outcomes attributable to the work |

Cover EACH dimension the target level requires. Use the ORG's actual
rubric where it exists; this is the generic shape.

## Activity → impact rewrite

| Activity (weak) | Impact (strong) |
|---|---|
| "Refactored auth" | "Refactored auth, eliminating a recurring outage class; new services onboard 2× faster" |
| "Led the migration" | "Led the X migration off legacy Y, cutting incident rate 40% and unblocking 3 teams" |
| "Mentored juniors" | "Mentored 2 engineers to promotion; raised the team's review quality (defect escape ↓)" |

Every line = outcome + measurable effect + scope of influence.

## Sustained-pattern test

- Is this ONE project or a repeated pattern of level-appropriate work?
- Would a skeptic see the level as the person's baseline, or a stretch?
- Are there multiple, independent examples per key dimension?

One spike reads as a stretch; a pattern reads as the new normal.

## Corroboration

- Peer/manager/stakeholder testimony.
- Artifacts: design docs, ADRs, dashboards/metrics, launched work.
- Cross-functional recognition.

Committees discount self-assertion; they trust corroborated impact.

## Gap-analysis format

```
Dimension coverage:
  Scope:        strong | adequate | THIN — <evidence / gap>
  Technical:    ...
  Influence:    ...
  Impact:       ...
Recommendation: ready | strengthen-then-submit (what) | not-yet (what's missing)
```

Name gaps honestly. A hidden gap the committee finds costs more than the
gap itself. Never inflate impact beyond the evidence.
